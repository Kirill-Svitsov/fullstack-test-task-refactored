# Рефакторинг бэкенда

## Цель

Провести рефакторинг бэкенд-части файлообменника, сохранив бизнес-логику, но улучшив архитектуру, читаемость и
поддерживаемость кода.

### Новая архитектура

Была применена чистая архитектура (Clean Architecture) с разделением на слои:

```
backend/src/
├── core/ # Ядро: конфигурация, подключение к БД
│ ├── config.py # Настройки приложения (переменные окружения)
│ └── database.py # Подключение к БД, фабрика сессий
│
├── domain/ # Доменный слой (бизнес-сущности)
│ ├── entities/
│ │ └── models.py # Модели SQLAlchemy (StoredFile, Alert)
│ └── repositories/
│ └── file_repository.py # Абстрактные интерфейсы репозиториев
│
├── application/ # Слой приложения (бизнес-логика)
│ ├── services/
│ │ ├── file_service.py # Логика работы с файлами
│ │ └── alert_service.py # Логика работы с алертами
│ └── dtos/ # Data Transfer Objects
│
├── infrastructure/ # Инфраструктурный слой (реализации)
│ ├── repositories/
│ │ └── postgres_repositories.py # Реализация репозиториев (PostgreSQL)
│ ├── storage/
│ │ └── local_storage.py # Работа с файловой системой
│ └── tasks/ # Celery задачи (будут позже)
│
└── interfaces/ # Интерфейсный слой (API, схемы)
├── api/
│ └── routes/
│ ├── files.py # Роутеры для файлов
│ └── alerts.py # Роутеры для алертов
└── schemas/
└── file_schemas.py # Pydantic схемы для валидации
```

## Ключевые изменения

### Описание слоев

- **core** — настройки и подключение к БД.
- **domain** — бизнес-сущности и интерфейсы репозиториев (не зависят от фреймворков).
- **application** — бизнес-логика (use cases).
- **infrastructure** — реализации репозиториев и внешних сервисов.
- **interfaces** — API, схемы, роутеры.

1. ### Разделение ответственности

   #### Было: Вся логика в одном файле (service.py) — и работа с БД, и бизнес-логика, и хранение файлов.

   #### Стало: Четкое разделение на слои: презентация (API) → бизнес-логика (сервисы) → доступ к данным (репозитории).

2. ### Внедрение зависимостей (Dependency Injection)

   #### Было: Глобальные объекты engine, async_session_maker.

   #### Стало: Зависимости передаются через конструкторы (FileService получает FileRepository и Storage). Это упрощает тестирование и замену реализаций.

3. ### Репозитории

   #### Было: Прямые запросы к БД в сервисах.

   #### Стало: Выделен слой репозиториев с четким интерфейсом (абстрактные классы). Это позволяет легко заменить БД или добавить кеширование.

4. ### Конфигурация

   #### Было: Переменные окружения разбросаны по коду.

   #### Стало: Единый класс Settings с валидацией и централизованным доступом ко всем настройкам.

5. ### Обработка ошибок

   #### Было: Исключения обрабатывались не везде.

   #### Стало: Добавлена единая обработка HTTP-исключений с понятными сообщениями.

6. ### Работа с файлами

   #### Было: Файлы сохранялись прямо в сервисе.

   #### Стало: Выделен отдельный класс LocalStorage для работы с файловой системой. Это позволяет легко заменить хранилище (например, на S3).

### Преимущества новой архитектуры

- **Single Responsibility** — каждый слой отвечает за свою задачу.
- **Open/Closed** — легко добавлять новые фичи без изменения существующего кода.
- **Dependency Inversion** — сервисы зависят от абстракций (репозиториев), а не от конкретных реализаций.
- **Тестируемость** — каждый слой можно тестировать изолированно.
- **Поддерживаемость** — легко найти и исправить ошибку, зная, где искать.
- **Масштабируемость** — легко добавить новые фичи (кеширование, новый тип хранилища).
- **Читаемость** — код структурирован, каждый файл отвечает за свою задачу.
- **Гибкость** — легко заменить реализацию (например, перейти с локального хранилища на S3).

### Исправленные баги

- **Trailing Slash** — в роутерах убраны лишние `/` в конце маршрутов, что устранило редиректы с 307 на корректные ответы 200/201.
- **Миграции** — обновлен `migrations/env.py`: импорт `DB_URL` заменен на `get_settings().DATABASE_URL`, исправлен импорт `Base` из `src.core.database`.
- **Импорты** — обновлены все пути после реструктуризации, удалены устаревшие импорты `src.service`, `src.models`.

### Проверка работоспособности

#### Создание файла

```
curl -X POST http://localhost:8000/files \
-F "title=Test File" \
-F "file=@/tmp/test.txt"
```

#### Ответ:

```
{
"id": "8c84a074-...",
"title": "Test File",
"original_name": "test.txt",
"size": 13,
"processing_status": "uploaded"
}
```

#### Получение списка файлов

```
curl http://localhost:8000/files
```

#### Возвращает массив с загруженными файлами

#### Получение алертов

```
curl http://localhost:8000/alerts
```

#### Возвращает массив алертов (пока пустой)
